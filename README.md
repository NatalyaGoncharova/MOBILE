# Android Page Object Contest Task

Задание для мануальных тестировщиков, которые хотят перейти в мобильную автоматизацию.

Нужно автоматизировать Android-сценарий покупки номера в приложении Tele2, используя `pytest`, `Appium` и паттерн `Page Object`.

## Цель

Проверить, что участник:

- понимает границы ответственности теста, page object и фикстур;
- умеет проектировать читаемые автотесты по ручному сценарию;
- не пишет проверки и локаторы прямо в тестовом методе;
- умеет работать с ожиданиями, состояниями экрана и навигацией;
- может автоматизировать неавторизованный flow;
- для extra-задания понимает разницу между авторизованной и неавторизованной зоной.

## Ограничения

- Только Android.
- Только бренд Tele2.
- iOS и Altel добавлять не нужно.
- Не нужно подключать Allure TestOps, BrowserStack, CI и приватные API.
- APK не хранится в репозитории. Положите файл локально в `app/tele2/android/app.apk`.
- Реальную покупку номера завершать не нужно: сценарий останавливается на экране регистрации и ввода ИИН.

## Быстрый старт

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
appium --port 4723
```

Заполните `.env`:

- `TELE2_PHONE` и `TELE2_PASSWORD` - пользователь для extra-сценария в авторизованной зоне;
- `ANDROID_APP_PATH` - путь до APK.

## Что нужно сделать

### 1. Реализовать Page Objects

Заполните page objects в `src/pages/`:

- `BecomeClientPage`
- `NumberPurchasePage`
- `RegistrationPage`
- `LoginPage` и `MainPage` - нужны для extra-задания в авторизованной зоне

В page object должны быть:

- локаторы;
- действия пользователя;
- проверки состояния экрана;
- ожидания элементов.

В тестах не должно быть прямого использования `driver.find_element`, `WebDriverWait`, `AppiumBy` и xpath/id локаторов.

Часть Android-локаторов для `Стать клиентом` / `Покупка номера` уже добавлена в `src/pages/locators.py` по текущему проекту. Если в актуальной сборке часть локаторов отличается, участник должен найти стабильную замену через APK, Appium Inspector или аналогичный инструмент.

### 2. Реализовать основной тест

#### TC-001. Покупка номера без авторизации до экрана ИИН

Тест: `test_unauthorized_user_can_reach_iin_step_from_number_purchase`

Предусловия:

- приложение открыто без авторизации;
- доступен каталог номеров для покупки.

Шаги:

1. Открыть приложение без авторизации.
2. Нажать `Стать клиентом`.
3. Выбрать город `Алматы`.
4. Проверить, что открылся экран выбора клиентского сценария.
5. Выбрать `Покупка номера`.
6. Проверить, что открылся экран покупки номера.
7. Выбрать любой доступный номер.
8. Продолжить оформление.
9. Дойти до экрана регистрации.
10. Проверить, что отображается выбранный номер.
11. Дойти до шага ввода ИИН.
12. Проверить, что поле ИИН отображается.

Ожидаемый результат:

- неавторизованный пользователь может начать покупку номера;
- выбранный номер сохраняется в оформлении;
- сценарий доходит до экрана регистрации и ввода ИИН.

### 3. Extra-задание

Дополнительно можно написать тест для раздела покупки номера в авторизованной зоне.

Возможное название:

```python
test_authorized_user_can_open_number_purchase_catalog
```

или, если сценарий должен идти дальше:

```python
test_authorized_user_can_start_number_purchase_flow
```

Минимальные ожидания:

1. Авторизоваться через `LoginPage`.
2. Из главной страницы открыть раздел покупки номера через `MainPage`.
3. Проверить, что открылся каталог номеров.
4. Выбрать доступный номер или проверить, что выбор доступен.
5. Проверить следующий экран или понятное ограничение, если авторизованному пользователю нельзя продолжить покупку.

Если extra-сценарий зависит от конкретного состояния аккаунта, это нужно описать в README или комментарии к тесту.

## Что оценивается

Подробная шкала оценки находится в [docs/evaluation.md](docs/evaluation.md).

Ключевые критерии:

- читаемость теста;
- корректная декомпозиция Page Objects;
- стабильные локаторы и ожидания;
- отсутствие sleep без причины;
- понятные assertions;
- минимальная дубликация;
- аккуратная работа с тестовыми данными;
- понимание различий между авторизованным и неавторизованным flow.

## Что сдать

Участник должен отправить ссылку на свой fork и merge request / pull request.

Fork - это личная копия этого репозитория в аккаунте участника. Работать нужно в своей копии, не в исходном репозитории с заданием.

### Как сдать через fork

1. Откройте репозиторий с заданием в GitLab или GitHub.
2. Нажмите `Fork`.
3. Склонируйте свой fork:

```bash
git clone <ссылка-на-ваш-fork>
cd MOBILE
```

4. Создайте отдельную ветку для решения:

```bash
git checkout -b solution/number-purchase
```

5. Выполните задание.
6. Закоммитьте изменения:

```bash
git add .
git commit -m "Implement number purchase test"
```

7. Отправьте ветку в свой fork:

```bash
git push origin solution/number-purchase
```

8. Создайте merge request / pull request из ветки `solution/number-purchase`.
9. Пришлите ссылку на merge request / pull request.

В MR/README нужно коротко описать:

- какой сценарий покрыт;
- какие Page Objects созданы;
- какие предположения были сделаны;
- выполнялось ли extra-задание;
- что не удалось реализовать и почему.
